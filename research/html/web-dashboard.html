Features

Memory Browser

Paginated table (20/page) with category filter dropdown. Click row for detail view. Columns: Key, Category, Importance, Tags, Updated.

Memory Detail

Full content display, category + importance badges, tags, metadata grid (ID, project, dates, access count, version). Delete with confirmation.

Semantic Search

Natural language query input. Results show key, match type (vector/keyword), relevance score (%), content preview (300 chars).

API Key Management

Create keys (name input), one-time display with copy button. Table: name, prefix, created, last used. Revoke with confirmation.

Serving Strategy

Dashboard builds to packages/dashboard/dist/. Server detects this directory at startup and serves static files. Route priority: /api/* and /mcp first, everything else falls through to static files with SPA fallback.

// In app.ts
const dashboardDist = resolve(__dirname, "../../dashboard/dist");
if (existsSync(dashboardDist)) {
  app.use("/*", serveStatic({ root: dashboardDist }));
  app.get("*", serveStatic({ root: dashboardDist,
    rewriteRequestPath: () => "/index.html" }));
}

Design Decisions

Why no UI framework? Dashboard is small (~6 components). Vanilla CSS with custom properties = zero dependency, 6.8 kB total. A framework would be larger.

Why same port? Zero extra config. No CORS. Single process. npm run dev → everything works. Dashboard is optional — if dist doesn’t exist, server still works.

Why localStorage for API key? Simple persistence. Dashboard is local-only (localhost), security risk is minimal. No cookies to manage.
