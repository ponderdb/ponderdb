1. System Overview

PonderDB is a local-first AI agent memory server with semantic search, REST API, MCP support, and a web dashboard.

3. Data Flow Diagrams

Memory Write Flow
sequenceDiagram
    participant C as Client
    participant A as Auth Middleware
    participant S as Memory Service
    participant E as Embedder
    participant DB as SQLite
    participant V as sqlite-vec

    C->>A: POST /api/memories (Bearer pndr_xxx)
    A->>A: Validate API key (SHA256 hash lookup)
    A->>S: Authorized request
    S->>S: Validate input + auto-detect category
    S->>E: Embed key + content
    E-->>S: 384-dim vector
    S->>DB: INSERT INTO memories
    S->>V: INSERT INTO vec_memories
    S-->>C: 201 Created
Hybrid Search Flow
sequenceDiagram
    participant C as Client
    participant S as Search Service
    participant E as Embedder
    participant V as sqlite-vec
    participant DB as SQLite

    C->>S: POST /api/memories/search
    S->>E: Embed query
    E-->>S: query vector

    par Parallel
        S->>V: KNN vector search
        V-->>S: semantic results
    and
        S->>DB: LIKE keyword search
        DB-->>S: keyword results
    end

    S->>S: Merge + deduplicate + sort
    S-->>C: Ranked results
MCP Remember Tool Flow
sequenceDiagram
    participant AI as AI Tool (Claude/Cursor)
    participant MCP as MCP Server
    participant Store as SQLite Store
    participant Embed as Embedder

    AI->>MCP: remember(key, content)
    MCP->>Store: getByKey(key)
    alt Exists
        MCP->>Embed: embed(key + content)
        MCP->>Store: update(id, content, embedding)
        MCP-->>AI: Updated key
    else New
        MCP->>MCP: detectCategory(content)
        MCP->>Embed: embed(key + content)
        MCP->>Store: create(key, content, category, embedding)
        MCP-->>AI: Remembered key (category)
    end

6. Authentication & Project Scoping

Auth Flow
sequenceDiagram
    participant C as Client
    participant MW as Auth Middleware
    participant DB as api_keys table

    C->>MW: Bearer pndr_xxxxx
    MW->>MW: SHA256 hash token
    MW->>DB: Lookup by key_hash
    alt Valid & not expired
        MW->>DB: Update last_used_at
        MW-->>C: Request allowed
    else Invalid
        MW-->>C: 401 Unauthorized
    end

How Project Scoping Works

  • API key authenticates the user
  • projectId parameter scopes memories to a project
  • UNIQUE(key, project_id) allows same key in different projects
  • SDK/MCP pass projectId with every operation
  • Dashboard has project selector filtering all views

8. Planned: Dynamic Categories

Problem

Categories are currently hardcoded as a fixed enum of 10 types. Users can't create custom categories, and AI can't generate new ones based on content patterns.

Planned: Dynamic Category Flow
flowchart TD
    A[New Memory Created] --> B{Category provided?}
    B -->|Yes| C{Category exists?}
    B -->|No| D[AI Auto-Detect]

    C -->|Yes| E[Use existing]
    C -->|No| F[Create new category]

    D --> G{Match existing?}
    G -->|Yes| E
    G -->|No| H[AI suggests new category]
    H --> K[Create AI-generated category]

    E --> L[Store Memory]
    F --> L
    K --> L

Security Architecture

API Key Security
graph TB
    REQ["Client Request"] --> |"Bearer pndr_xxx"| MW[Auth Middleware]
    MW --> HASH["SHA256(token)"]
    HASH --> DB[(api_keys table)]
    DB --> CHECK{Valid?}
    CHECK --> |Yes| UPDATE["Update last_used_at"]
    CHECK --> |No| REJECT["401 Unauthorized"]
    UPDATE --> ALLOW["Request Proceeds"]

Security Properties

  • Keys never stored in plaintext (SHA256 hashed)
  • Format: pndr_ prefix for identification
  • Auto-generated on first server start, printed once
  • last_used_at tracking for audit
  • Optional expiration support
